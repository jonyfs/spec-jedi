# Spec Jedi-তে অবদান রাখা

Spec Jedi তার নিজস্ব [constitution](../../../.specify/memory/constitution.md)-এর
অধীনে তৈরি — অলঙ্ঘনীয় নিয়মের একটি ভার্সনযুক্ত সেট যার বিপরীতে প্রতিটি পরিবর্তন,
এটি-সহ, যাচাই করা হয়। এই ডকুমেন্টটি "আমি আসলে কীভাবে অবদান রাখব" এর ব্যবহারিক সঙ্গী;
constitution হলো প্রতিটি নিয়মের অস্তিত্বের *কেন*-এর চূড়ান্ত বিবৃতি।

## কিছু লেখার আগে

1. **Constitution পড়ুন।** কমপক্ষে, [Core Principles](../../../.specify/memory/constitution.md)
   I-XX দেখে নিন — অধিকাংশ অবদান সংক্রান্ত প্রশ্নের ("আমার কি টেস্ট দরকার", "এটি কীভাবে
   নামকরণ করা উচিত", "এর জন্য আগে গবেষণা দরকার কি না") উত্তর সেখানে ইতিমধ্যে দেওয়া আছে।
2. **যেকোনো নতুন `specjedi-*` স্কিলের জন্য প্রতিযোগিতামূলক গবেষণা আবশ্যক**
   (Principle II, আলোচনার অযোগ্য)। একটি নতুন স্কিল প্রস্তাব করার আগে, এটিকে
   [github/spec-kit](https://github.com/github/spec-kit) এবং কমপক্ষে অন্য দশটি
   সর্বজনীনভাবে উপলব্ধ SDD টুলের সাথে বেঞ্চমার্ক করুন, এবং কমপক্ষে একটি প্রকৃত অবদানের
   কথা বলুন যা আপনার প্রস্তাব তাদের ইতিমধ্যে যা প্রদান করে তার বাইরে দেয়। এটি ফিচার
   স্পেকের পাশে একটি `research.md`-এ লিখুন — প্রত্যাশিত আকারের জন্য দেখুন
   `specs/001-specjedi-pipeline/research.md` এবং `specs/002-specjedi-onboard/research.md`।
3. **`references/skill-roadmap.md` পরীক্ষা করুন** — আপনার আইডিয়া হয়তো ইতিমধ্যে
   অগ্রাধিকার নোটসহ সেখানে সীমাবদ্ধ করা আছে; একটি বিদ্যমান প্রস্তাব সম্প্রসারণ করা
   সাধারণত একটি প্রতিদ্বন্দ্বী প্রস্তাব খোলার চেয়ে ভালো।

## পরিবর্তনগুলো কীভাবে ship হয়

এই প্রজেক্ট trunk-based (Principle X):

- `main` হলো trunk। **কোনো কমিট কখনো সরাসরি `main`-এ যায় না।**
- প্রতিটি পরিবর্তন একটি pull request হিসেবে তার নিজস্ব স্বল্পস্থায়ী ব্রাঞ্চে বের হয়।
- CI (`ci-gate`) প্রতিটি PR-এ সম্পূর্ণ validation battery চালায় — structural lint,
  cross-platform চেক (Linux/macOS/Windows, Principle XIII)। একটি PR তখনই merge হয় যখন
  প্রতিটি required চেক green হয়; কোনো manual override বা "যেভাবেই হোক merge করো" পথ
  নেই।
- **শুধুমাত্র চেকের ভিত্তিতে অটো-মার্জ হলো repository owner-এর নিজস্ব PR-এর
  একটি বিশেষাধিকার।** আপনি যদি একজন বহিরাগত অবদানকারী হন, তাহলে merge হওয়ার আগে green
  `ci-gate`-এর পাশাপাশি owner-এর কাছ থেকে একটি স্পষ্ট APPROVED review আপনার PR-এর
  প্রয়োজন — সঠিক প্রক্রিয়ার জন্য `.github/workflows/validate.yml`-এ `owner-gate` job
  দেখুন।

## একটি `specjedi-*` স্কিল যোগ করা বা পরিবর্তন করা

নতুন স্কিল এবং বিদ্যমান স্কিলে উল্লেখযোগ্য পরিবর্তন এই প্রজেক্টের নিজস্ব SDD পাইপলাইন
অনুসরণ করে — Spec Jedi প্রোডাক্ট হিসেবে যা ship করে তার একই (constitution-এর
Development Workflow বিভাগ):

1. **গবেষণা** (Principle II) — উপরে দেখুন।
2. **নির্দিষ্টকরণ (Specify)** — অগ্রাধিকারভিত্তিক ইউজার স্টোরি, কার্যকরী প্রয়োজনীয়তা,
   এবং পরিমাপযোগ্য সাফল্যের মানদণ্ডসহ একটি `spec.md`। অনুমান করার বদলে প্রকৃত
   অস্পষ্টতাকে `NEEDS CLARIFICATION` দিয়ে চিহ্নিত করুন (Principle V)।
3. **স্পষ্টীকরণ (Clarify)** — অনুমানের ভিত্তিতে পরিকল্পনা করার আগে চিহ্নিত যেকোনো
   অস্পষ্টতা সমাধান করুন।
4. **পরিকল্পনা (Plan)** — একটি প্রকৃত Constitution Check গেটসহ একটি `plan.md`: যদি
   আপনার পরিবর্তন কোনো নীতি লঙ্ঘন করে, তাহলে হয় এটি সরলীকরণ করুন অথবা Complexity
   Tracking-এ স্পষ্টভাবে যুক্তি লিপিবদ্ধ করুন, কখনো নীরবে গেট পাস করবেন না।
5. **টাস্ক (Tasks)** — ডিপেন্ডেন্সি অনুসারে ক্রমানুসারে সাজানো একটি `tasks.md`, যেখানে
   পরিকল্পনায় কোড প্রয়োজন সেখানে টেস্ট-প্রথম (Principle VI)।
6. **বাস্তবায়ন (Implement)** — শুধুমাত্র একটি ফিচার ব্রাঞ্চ এবং PR-এর মাধ্যমে (উপরে
   দেখুন), কখনো trunk-এ সরাসরি কমিট নয়।

এই রিপোজিটরিতে প্রকাশিত প্রতিটি `specs/NNN-feature/` ডিরেক্টরি এই আকারের একটি
কার্যকরী উদাহরণ — `specs/001-specjedi-pipeline/` এবং `specs/002-specjedi-onboard/`
সবচেয়ে সম্পূর্ণ রেফারেন্স।

## স্কিল রচনা এবং প্রম্পট ইঞ্জিনিয়ারিং স্ট্যান্ডার্ড

এই রিপোজিটরির প্রতিটি `SKILL.md`, নতুন হোক বা পরিবর্তিত, অবশ্যই অন্তর্ভুক্ত করতে হবে
(Principle XIX; সম্পূর্ণ বিস্তারিত
[`references/skill-authoring-standard.md`](../../../references/skill-authoring-standard.md)-এ):

- একটি স্পষ্ট persona এবং task বিবৃতি।
- একটি সংজ্ঞায়িত আউটপুট ফরম্যাট।
- কমপক্ষে একটি সম্পূর্ণ "ইনপুট → আউটপুট" কার্যকরী উদাহরণ।
- যেকোনো non-deterministic বিচারের জন্য chain-of-thought নির্দেশনা।
- স্পষ্ট **Always** / **Never** গার্ডরেল।
- যাচাইযোগ্য সাফল্যের মানদণ্ড — যাচাইযোগ্য তথ্য, অনুভূতি নয়।
- যেখানে স্কিলের নিজস্ব বর্ণনার প্রয়োজন সেখানে দর্শক ক্যালিব্রেশন (নতুন থেকে উন্নত
  পর্যন্ত, Principle I)।

## পর্যালোচনার অনুরোধের আগে যাচাইকরণ

একটি PR খোলার আগে স্থানীয়ভাবে structural lint চালান:

```bash
bash scripts/validate.sh      # macOS/Linux
pwsh scripts/validate.ps1     # Windows
```

উভয়কেই পাস করতে হবে — এই প্রজেক্ট সমানভাবে Linux, macOS, এবং Windows সমর্থন করে
(Principle XIII); এমন একটি পরিবর্তন যা শুধুমাত্র একটি প্ল্যাটফর্মে কাজ করে তা সম্পূর্ণ
নয়।

আপনার স্কিল যদি কোড তৈরি করে, তাহলে এর elicitation প্রশ্ন এবং branching লজিক
ডকুমেন্টেড অনুযায়ী আচরণ করে তা নিশ্চিত করতে একটি scenario-based dry run-ও প্রয়োজন
(Principle IX) — PR বিবরণে আপনি কী পরীক্ষা করেছেন তা বর্ণনা করুন।

## কণ্ঠস্বর এবং নামকরণ

- প্রোডাক্ট স্কিলের নাম `specjedi-*` দেওয়া হয়, কখনো `speckit-*` নয় — পরেরটি হলো
  vendored অভ্যন্তরীণ bootstrap টুলিং যা এই রিপোজিটরি নিজেকে তৈরি করতে ব্যবহার করে, এটি
  এমন কিছু নয় যা end user ইনস্টল করে (Principle XV)। এই পার্থক্য স্পষ্ট না হলে README-এর
  "Spec Jedi কীভাবে নিজেকে তৈরি করে" বিভাগ দেখুন।
- End-user-facing বর্ণনা (জেনারেটেড `spec.md`/`plan.md`/`tasks.md` ক্ষেত্রের আক্ষরিক
  বিষয়বস্তু নয়) Spec Jedi-এর Star Wars-flavored কণ্ঠস্বর ব্যবহার করে (Principle XII) —
  রেফারেন্স লেক্সিকনের জন্য `references/star-wars-lexicon.md` দেখুন। জেনারেটেড
  আর্টিফ্যাক্টের *বিষয়বস্তু* সঠিক এবং পরিভাষা-উপযুক্ত থাকে; কণ্ঠস্বর তার চারপাশের
  স্কিলের নিজস্ব বর্ণনায় প্রযোজ্য।

## README-কে সৎ রাখা

যদি আপনার পরিবর্তন একটি প্রকাশিত স্কিল, একটি নতুন badge-worthy তথ্য যোগ করে, বা অন্য
কোনোভাবে প্রজেক্ট সম্পর্কে যা সত্য তা পরিবর্তন করে, তাহলে একই PR-এ `README.md` আপডেট
করুন — "আজ আপনি কী পাচ্ছেন" টেবিল, Quickstart নম্বরিং, এবং পাইপলাইন Mermaid ডায়াগ্রাম
সবকিছুই সিঙ্কে থাকতে হবে (Principle XVI)। PR খোলার আগে badge সারি পর্যালোচনা করুন:
নিশ্চিত করুন প্রতিটি বিদ্যমান badge এখনও সঠিকভাবে দেখাচ্ছে, এবং শুধুমাত্র তখনই একটি নতুন
badge যোগ করুন যখন আপনার পরিবর্তন সত্যিকারের নতুন, প্রাসঙ্গিক তথ্য যা সংকেত দেওয়ার
যোগ্য — কখনো এমন একটি hand-typed মান নয় যা পুরনো হয়ে যেতে পারে (Distribution &
Ecosystem Standards)।

## প্রশ্ন

যোগ্য — এমন একটি নিয়ম যা কেউ অনুসরণ করতে পারে না কারণ এটি অস্পষ্ট, তা constitution-এর
একটি ত্রুটি, আপনার নয়।
