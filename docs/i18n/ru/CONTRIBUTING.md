# Участие в Spec Jedi

Spec Jedi построен на основе собственной
[constitution](../../../.specify/memory/constitution.md) — версионного
набора неотъемлемых правил, по которым проверяется каждое изменение,
включая это. Этот документ — практический спутник вопроса "как я на
самом деле участвую"; constitution — окончательное заявление о том,
*почему* существует каждое правило.

## Прежде чем что-либо писать

1. **Прочитайте constitution.** Как минимум, просмотрите
   [Core Principles](../../../.specify/memory/constitution.md) I-XX —
   на большинство вопросов об участии ("нужны ли мне тесты", "как это
   должно называться", "нужно ли этому сначала исследование") там уже
   есть ответы.
2. **Конкурентное исследование обязательно для любого нового навыка
   `specjedi-*`** (Principle II, не подлежит обсуждению). Прежде чем
   предлагать новый навык, сравните его с
   [github/spec-kit](https://github.com/github/spec-kit) и как минимум
   десятью другими публично доступными инструментами SDD, и назовите
   как минимум один подлинный вклад, который ваше предложение делает
   сверх того, что уже предлагает любой из них. Запишите это в
   `research.md` рядом со спецификацией функции — см.
   `specs/001-specjedi-pipeline/research.md` и
   `specs/002-specjedi-onboard/research.md` для ожидаемой формы.
3. **Проверьте `references/skill-roadmap.md`** — ваша идея, возможно,
   уже там намечена с заметками о приоритизации; расширение
   существующего предложения обычно лучше, чем открытие конкурирующего.

## Как поставляются изменения

Этот проект основан на trunk (Principle X):

- `main` — это ствол. **Ни один коммит никогда не попадает напрямую в
  `main`.**
- Каждое изменение выходит на собственной недолговечной ветке в виде
  pull request.
- CI (`ci-gate`) запускает всю батарею валидации — структурный lint,
  кроссплатформенные проверки (Linux/macOS/Windows, Principle XIII) —
  на каждом PR. PR сливается только после того, как каждая обязательная
  проверка станет зелёной; нет ручного обхода или пути "слить в любом
  случае".
- **Автослияние, основанное только на проверках, — привилегия
  собственных PR владельца репозитория.** Если вы внешний участник,
  вашему PR требуется явное APPROVED-ревью от владельца в дополнение к
  зелёному `ci-gate` перед слиянием — см. задачу (job) `owner-gate` в
  `.github/workflows/validate.yml` для точного механизма.

## Добавление или изменение навыка `specjedi-*`

Новые навыки и существенные изменения существующих навыков следуют
собственному пайплайну SDD этого проекта — тому же, который Spec Jedi
поставляет как продукт (раздел Development Workflow constitution):

1. **Исследование** (Principle II) — см. выше.
2. **Спецификация** — `spec.md` с приоритизированными
   пользовательскими историями, функциональными требованиями и
   измеримыми критериями успеха. Отмечайте подлинную неоднозначность с
   помощью `NEEDS CLARIFICATION` вместо угадывания (Principle V).
3. **Уточнение** — разрешите любую отмеченную неоднозначность прежде,
   чем планировать на основе догадки.
4. **Планирование** — `plan.md` с настоящим гейтом Constitution Check:
   если ваше изменение нарушило бы принцип, либо упростите его, либо
   явно зафиксируйте обоснование в Complexity Tracking, никогда не
   проходите гейт молча.
5. **Задачи** — `tasks.md`, упорядоченный по зависимостям, сначала
   тесты там, где план требует кода (Principle VI).
6. **Реализация** — только через feature-ветку и PR (см. выше), никогда
   прямой коммит в ствол.

Каждый поставленный каталог `specs/NNN-feature/` в этом репозитории —
рабочий пример этой формы — `specs/001-specjedi-pipeline/` и
`specs/002-specjedi-onboard/` являются наиболее полными эталонами.

## Стандарт авторства навыков и prompt engineering

Каждый `SKILL.md` в этом репозитории, новый или изменённый, ДОЛЖЕН
включать (Principle XIX; полные детали в
[`references/skill-authoring-standard.md`](../../../references/skill-authoring-standard.md)):

- Чёткую персону и формулировку задачи.
- Определённый формат вывода.
- Как минимум один полный проработанный пример "вход → выход".
- Инструкцию по цепочке рассуждений (chain-of-thought) для любого
  недетерминированного решения суждения.
- Явные ограждения **Always** / **Never**.
- Проверяемые критерии успеха — проверяемые факты, а не ощущения.
- Калибровку аудитории там, где это нужно собственному повествованию
  навыка (от новичка до продвинутого, Principle I).

## Валидация перед запросом ревью

Запустите структурный lint локально перед открытием PR:

```bash
bash scripts/validate.sh      # macOS/Linux
pwsh scripts/validate.ps1     # Windows
```

Оба должны пройти — этот проект одинаково поддерживает Linux, macOS и
Windows (Principle XIII); изменение, работающее только на одной
платформе, не завершено.

Если ваш навык производит код, ему также нужен dry run на основе
сценариев, подтверждающий, что его вопросы для выяснения и логика
ветвления ведут себя так, как задокументировано (Principle IX) —
опишите, что вы проверили, в описании PR.

## Голос и наименование

- Продуктовые навыки называются `specjedi-*`, никогда `speckit-*` —
  последнее — это поставляемая (vendored) внутренняя инфраструктура
  bootstrap, которую этот репозиторий использует для собственного
  построения, а не то, что устанавливают конечные пользователи
  (Principle XV). См. раздел README "Как Spec Jedi строит сам себя",
  если это различие неясно.
- Повествование, ориентированное на конечного пользователя (не
  буквальное содержимое сгенерированных полей `spec.md`/`plan.md`/
  `tasks.md`), использует собственный голос Spec Jedi в духе Star Wars
  (Principle XII) — см. `references/star-wars-lexicon.md` для
  справочного лексикона. *Содержимое* сгенерированного артефакта
  остаётся точным и уместным с точки зрения терминологии; голос
  применяется к собственному повествованию навыка вокруг него.

## Поддержание честности README

Если ваше изменение добавляет поставленный навык, новый факт,
достойный badge, или иным образом меняет то, что верно о проекте,
обновите `README.md` в том же PR — таблица "Что вы получаете сегодня",
нумерация Quickstart и диаграмма Mermaid пайплайна должны оставаться
синхронизированными (Principle XVI). Просмотрите строку badge'ей перед
открытием PR: убедитесь, что каждый существующий badge всё ещё
корректно отображается, и добавляйте новый только если ваше изменение
— это подлинно новый, релевантный факт, заслуживающий сигнала — никогда
не значение, написанное вручную, которое может устареть (Distribution &
Ecosystem Standards).

## Вопросы

Если что-то в constitution неясно, это стоит поднять как отдельную
issue — правило, которому никто не может следовать, потому что оно
неоднозначно, — это дефект constitution, а не ваш.
